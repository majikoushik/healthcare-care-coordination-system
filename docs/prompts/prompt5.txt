Epic 4 Goal:

Allow users to create, view, update, and manage synthetic patient care plans using Cosmos DB-ready document persistence.

Care Plan Management should demonstrate flexible healthcare document modeling, patient-centered coordination, follow-up planning, and polyglot persistence.

Care plans should be stored in Cosmos DB or through a Cosmos DB-ready abstraction. SQL Server should remain the persistence choice for Patient, Provider, and Appointment master/transactional data.

Validation rules:

* Patient ID is required.
* Patient ID must refer to an existing patient if patient lookup is available.
* Provider ID is required.
* Provider ID must refer to an existing provider if provider lookup is available.
* Title is required.
* Title should have a reasonable length limit.
* Clinical summary is required but must be synthetic demo text only.
* Instructions are required.
* Follow-up date should not be in the past for active care plans.
* Status must be a supported value.
* Goals collection may be empty initially but should be supported.
* Tasks collection may be empty initially but should be supported.
* Task due date should not be in the past for new pending tasks.
* Created timestamp should be system generated.
* Updated timestamp should be system generated or updated automatically.

Recommended care plan status transition rules:

* Draft → Active
* Draft → Cancelled
* Active → OnHold
* Active → Completed
* Active → Cancelled
* OnHold → Active
* OnHold → Cancelled
* Completed → no further transition
* Cancelled → no further transition

Care plans can contain sensitive health information. This project must use only synthetic demo content and must not claim production medical compliance.

Ensure implementation and documentation state:

1. Only synthetic care plan data is used.
2. This is not a production electronic health record system.
3. The project does not claim HIPAA compliance or medical certification.
4. Clinical summary and instructions must not contain real patient data.
5. Logs should use safe metadata such as care plan ID, patient ID, provider ID, and correlation ID.

7. Cosmos DB documents may contain healthcare-like data, so secure configuration and access-control readiness must be documented.
8. Future authentication and role-based access control should be documented.
